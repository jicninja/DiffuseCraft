# undo-redo-system — Requirements

> **Status:** Draft v0.1 — cross-cutting capability spec.
> **Depends on:** `mcp-tool-catalog` (`undo`/`redo` tools, `diffusecraft://undo-stack/...` resources), `server-architecture` (UndoRedoManager wiring into dispatcher), `canvas-fundamentals` (the document state being versioned).
> **References:** P27 (Universal undo/redo — non-negotiable), P1 (agent-first), P5 (state queryable), `mcp-tool-catalog` FR-22/23 (per-client per-document, multi-client conflict semantics).

## 1. Purpose

Define the **universal undo/redo system** every editor spec wires into. This is **cross-cutting capability**, not a feature: every state-mutating operation that ships in v1 must register a `Command` so it is reversible. Without this, `apply_history_item`, `add_layer`, `paint_strokes`, every transform, every region/control-layer mutation become irreversible — unusable for both humans and agents.

This spec defines:
- The `Command` interface and the rules every reversible handler obeys.
- Per-client per-document stack semantics (each token has its own stacks per document).
- Multi-client merge + conflict resolution.
- Snapshot policy (every N ops, full document snapshot).
- Memory and storage budgets.
- The MCP surface (`undo`, `redo`, undo-stack/redo-stack resources).
- Tablet UX integration.
- Boundaries: what is reversible, what isn't, why.

## 2. Stakeholders & user stories

### S1 — Illustrator iterating on the canvas
> **Story 1.** As an illustrator, I add a layer, paint on it, transform it, generate a fill, apply the result, draw on top. I two-finger tap several times and each step reverts in order: drawing → fill apply → transform → paint → layer add. Three-finger tap re-applies them.

### S2 — Two clients on one document
> **Story 2.** As a user with my tablet AND Claude Code paired to the same server, both editing the same document: I add a paint layer (operation by my tablet token); Claude adds a control layer (operation by Claude's token). I two-finger tap → my paint layer reverts; Claude's control layer is untouched. Claude calls `undo` → its control layer reverts; my paint layer is untouched.

### S3 — Agent recovering from a mistake
> **Story 3.** As Claude Code that just made a wrong call (e.g., `apply_history_item` on the wrong preview), I call `undo()`. The mistaken layer disappears. I retry with the correct preview.

### S4 — Long session with many ops
> **Story 4.** As an illustrator with a 200-op session, the undo stack holds the last 100 ops (default depth). Older ops are dropped from the stack but their effects remain on the canvas. The system uses a snapshot every 20 ops to keep restore times bounded.

### S5 — Crash recovery
> **Story 5.** As a user whose tablet app crashed mid-operation, I reopen the app. The document is intact (server-side persistence). The undo stack is **gone** (intentionally — stacks are session-scoped). I can't undo what I did before the crash, but the canvas state is correct.

## 3. Functional requirements (EARS)

### 3.1 The Command pattern

**FR-1 (Ubiquitous).** Every reversible tool handler SHALL produce a `Command` with:

```typescript
interface Command {
  id: string;                                     // ULID
  tool_name: string;                              // for audit/debug
  document_id: DocumentId;
  args_summary: string;                           // human-readable hint for UI ("Add layer 'Generated: red barn'")
  apply(): Promise<unknown>;                      // (re)applies the operation
  revert(): Promise<void>;                        // restores prior state
  apply_result?: unknown;                          // captured output for redo to return same payload
  weight: "small" | "medium" | "large";           // hints snapshot policy
}
```

**FR-2 (Ubiquitous).** `apply()` and `revert()` SHALL be **deterministic** for a given starting document state. Re-applying after revert MUST produce the same observable outcome (modulo non-deterministic seeds; for AI generation, re-applying replays the generated bytes from the history item, NOT regenerates).

**FR-3 (Ubiquitous).** A handler that is marked `reversible: true` in the catalog SHALL produce a Command; one marked `reversible: false` SHALL NOT. Build-time check enforces consistency.

### 3.2 Per-client per-document stacks (FR-22 of mcp-tool-catalog)

**FR-4 (Ubiquitous).** The `UndoRedoManager` SHALL maintain a separate **undo stack** and **redo stack** per `(token_id, document_id)` pair.

**FR-5 (Ubiquitous).** When client A invokes a reversible tool on document D, the resulting Command SHALL be pushed onto A's undo stack for D. Client B's stacks SHALL NOT be affected.

**FR-6 (Ubiquitous).** `undo` invoked by client A pops A's last Command from A's undo stack for the active document, calls `revert()`, then pushes the Command onto A's redo stack.

**FR-7 (Ubiquitous).** `redo` invoked by client A pops A's last Command from A's redo stack, calls `apply()`, then pushes onto A's undo stack.

**FR-8 (Ubiquitous).** Pushing a fresh Command onto A's undo stack SHALL **clear** A's redo stack for that document — the standard "branched edit" semantics.

**FR-9 (Ubiquitous).** Default stack depth: 100 Commands per (token, document). Configurable per server via `ServerConfig.undo.max_depth_per_client`.

### 3.3 Snapshots (bounded restore time)

**FR-10 (Ubiquitous).** Every N Commands (default `N=20`, configurable) registered to a (token, document) stack, the manager SHALL **persist a full document snapshot** in memory keyed by the Command at that point. Snapshots are anchors that bound revert chain length.

**FR-11 (Ubiquitous).** Snapshot creation SHALL be incremental: store the diff between the prior snapshot and the current state if cheaper than full snapshot. v1 may use full snapshots and revisit if memory is an issue.

**FR-12 (Ubiquitous).** Snapshots SHALL be discarded when their anchor Command falls out of the undo stack (depth limit reached).

### 3.4 Multi-client conflict resolution (FR-23 of mcp-tool-catalog)

**FR-13 (Event-driven).** WHEN client A and client B perform overlapping operations (touching same layer pixels OR modifying same property), THE server SHALL apply them in arrival order (last-write-wins) AND emit `document.changed { conflict: true }`.

**FR-14 (Ubiquitous).** Both Commands remain in their originating clients' undo stacks. Client A's `undo` reverts A's Command; if B's Command was applied on top, the revert may produce a state inconsistent with B's prior expectation — that's accepted multi-client behavior, signaled by the conflict flag.

**FR-15 (Ubiquitous).** No "merge" semantics in v1. Operations are sequential at the server; conflict means "B happened after A, so reverting A might not get back to clean state if B touched the same area."

### 3.5 Reversibility scope

**FR-16 (Ubiquitous).** **REVERSIBLE in v1**:
- All layer ops: add, remove, update (visibility, opacity, blend, name, position), duplicate, merge_down, flatten_visible.
- Group ops: create_group, ungroup, update_group, move_layers_into_group.
- Selection ops: set, clear, modify.
- Apply / discard history item.
- Add / remove control layer.
- Define / remove region.
- All paint ops: paint_strokes, paint_area.
- Set workspace.
- Set active document, set active layer.
- All transform ops (post `transform-tools` spec).

**FR-17 (Ubiquitous).** **NOT REVERSIBLE in v1** (deliberate):
- `generate_image` — submits a job; the job's result (a history item) is reversible via apply/discard.
- `cancel_job` — cancellation itself is final.
- `download_model`, `delete_model` — file-system ops outside document state.
- `set_preset`, `delete_preset` — server-config ops, not document state.
- `revoke_token`, `rotate_my_token` — security ops.
- `transcribe_audio`, `enhance_prompt` — return strings without mutating document.
- `upload_blob` — produces a ULID; no document state change.

**FR-18 (Ubiquitous).** Tools marked non-reversible SHALL document this in their MCP description so agents can plan accordingly.

### 3.6 MCP tools and resources

**FR-19 (Ubiquitous).** `undo({ document_id?: DocumentId })` (write, idempotent) — pops + reverts the last Command from calling client's stack on the specified or active document. Returns `{ reverted_command_id, args_summary }` or `{ no_op: true }` if stack empty.

**FR-20 (Ubiquitous).** `redo({ document_id?: DocumentId })` — analogous on the redo stack.

**FR-21 (Ubiquitous).** Resource `diffusecraft://undo-stack/<document_id>` returns `Paginated<CommandSummary>` for the calling token's undo stack on that document. Each entry: `id`, `tool_name`, `args_summary`, `created_at`. (No `apply`/`revert` exposed.)

**FR-22 (Ubiquitous).** Resource `diffusecraft://redo-stack/<document_id>` analogous.

**FR-23 (Event-driven).** WHEN `undo` or `redo` succeeds, THE server SHALL emit `document.changed { change_summary: "Undid: <args_summary>" | "Redid: ...", originating_token_name }` so other clients refresh.

### 3.7 Persistence

**FR-24 (Ubiquitous).** Undo / redo stacks SHALL be **in-memory only**, scoped to the current server session. Server restart clears all stacks; document content persists in SQLite.

**FR-25 (Ubiquitous).** Stack discard rules:
- On client disconnect (token's last connection closed): retain stacks for `ServerConfig.undo.retain_after_disconnect_seconds` (default 600 s = 10 min) so transient reconnects don't lose state. After timeout, discard.
- On document close (`set_active_document` to a different doc, or `close_document`): retain (user might come back).
- On token revocation: discard immediately for that token.
- On server `stop()`: discard (no persistence by design).

**FR-26 (Ubiquitous).** A future "persistent undo" feature could store stacks in SQLite; explicitly NOT v1.

### 3.8 Memory budget

**FR-27 (Ubiquitous).** Total undo memory across all (token, document) pairs SHALL be bounded by `ServerConfig.undo.max_total_memory_bytes` (default 512 MB). On overflow, evict oldest snapshots first; if still over budget, evict oldest Commands (start dropping from the **bottom** of the deepest stacks).

**FR-28 (Event-driven).** WHEN eviction occurs, THE server SHALL emit `undo.eviction { token_id, document_id, ops_evicted }` for observability.

**FR-29 (Ubiquitous).** Per-Command memory budget: a Command's `revert` payload SHALL be ≤ 16 MB (matches per-call payload cap from `mcp-tool-catalog` Q7-bis). Larger ops use snapshot-based revert (full snapshot > 16 MB but compressed and shared).

### 3.9 Tablet UX integration

**FR-30 (Ubiquitous).** Two-finger tap on canvas → `undo`. Three-finger tap → `redo` (already specced in `canvas-fundamentals` FR-28).

**FR-31 (Ubiquitous).** A toast SHALL appear briefly with `args_summary` of the reverted/redone op for ~1.5 s.

**FR-32 (Ubiquitous).** A panel (under "History" or "Edits") MAY show the undo stack as a list; user can tap an entry to revert to that state (multiple `undo` calls). Optional in v1.

**FR-33 (Ubiquitous).** "Edited remotely" indicator (per `canvas-fundamentals` FR-31) appears when a remote client's op affects a local layer; it does NOT alter the local user's undo stack.

### 3.10 Cross-cutting handler contract

**FR-34 (Ubiquitous).** Every handler in `libs/server/src/lib/handlers/*` for a reversible tool SHALL use the standard pattern:

```typescript
export const myHandler: Handler<typeof myTool> = async (input, ctx) => {
  // 1. Validate input
  // 2. Compute result + revert payload
  const command: Command = {
    id: ulid(),
    tool_name: myTool.name,
    document_id: input.document_id,
    args_summary: `${myTool.title}: ${summarize(input)}`,
    apply: async () => { /* the mutation, returns result */ },
    revert: async () => { /* restore */ },
    weight: "small",
  };
  return ctx.undoRedo.execute(ctx.tokenName, input.document_id, command);
};
```

**FR-35 (Ubiquitous).** The `reversibleCommandMw` middleware in `server-architecture` D.9 SHALL bridge non-reversible handlers (which return raw results) and reversible handlers (which return through `undoRedo.execute`).

## 4. Non-functional requirements

**NFR-1 (Ubiquitous).** `undo` invocation SHALL complete in < 100 ms for a small Command (e.g., layer visibility toggle); ≤ 500 ms for medium (e.g., layer add); ≤ 2 s for a large op restored from snapshot.

**NFR-2 (Ubiquitous).** Memory eviction events SHALL NOT block tool dispatch — eviction runs asynchronously.

**NFR-3 (Ubiquitous).** Stack maintenance overhead SHALL be O(1) for push/pop, O(log N) for snapshot lookup.

## 5. Out of scope

- **Persistent undo across server restart** — explicitly v2+.
- **Branching undo (alternate timelines)** — keeping multiple redo lines instead of clearing on a fresh op. Not v1.
- **Undo of non-document ops** (presets, models, tokens) — these are global state, not per-document; out of scope.
- **Cross-document undo** — undo always targets one document.

## 6. Open questions

### Q1 — Should the redo stack persist across `undo → fresh op` boundary as a "side branch"?
Some editors (Krita, Photoshop with "non-linear undo") allow branching.

**Recommendation:** **no in v1.** Standard `clear redo on new op` semantics. Branching is post-v1 (would require tree-based stack model — complex).

### Q2 — How is `apply_history_item`'s revert handled when the source preview was discarded later?
The history_item still exists in DB but `discarded_at` is set; the layer it created lives in the document.

**Recommendation:** **revert works regardless of discarded_at.** The Command captures the layer state, not a reference to the history item's status. Reverting an applied item un-creates the layer; the history_item's `discarded_at` is irrelevant to the reversal.

### Q3 — `clear_undo_stack` tool?
`mcp-tool-catalog` previously had `clear_undo_stack`; we removed it from v1 catalog. Confirm.

**Recommendation:** **stay removed.** Use cases are rare; equivalent achievable via `set_active_document` swap-and-back. Keeps catalog small.

### Q4 — Snapshot frequency tuning per document size
Larger documents → snapshots cost more memory; might want `N` to scale.

**Recommendation:** **fixed N=20 in v1.** If memory pressure arises in real-world testing, make adaptive.

### Q5 — Two clients editing rapidly: O(N) memory per stack with default 100 ops × 2 clients × 2 stacks
With heavy use, memory grows quickly.

**Recommendation:** memory budget (FR-27) handles overflow. If 2 clients × 100 ops × 2 stacks × ~5 MB avg = 2 GB → far over default 512 MB → eviction kicks. Acceptable; eviction logged for observability.

### Q6 — Undo of a transform applied to a Z-stack of layers via grouping
If a user transforms a group, the revert needs to restore all child layer transforms.

**Recommendation:** **group transform = single Command** with revert that restores all children. Per `transform-tools` later spec; mentioned here for cross-reference.

## 7. Acceptance criteria

This spec is APPROVED when:

1. The five user stories (§2) are realized.
2. The Command interface (§3.1) is implementable for every reversible tool listed in §3.5.
3. Per-client per-document stack semantics are unambiguous, including multi-client conflict (§3.4).
4. Memory budget (§3.8) is enforceable.
5. Tablet UX integration (§3.9) covers the canonical undo gesture and the "edited remotely" non-interference rule.
6. Open questions have acceptable recommendations.
